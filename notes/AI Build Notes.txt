2) A linux server on the LAN
2,3) Two full rsynced backups will be retained on the server ((C)current and (O)once removed).  The remote utility will first inform the server a new backup is about to be made.  The server will create a set of diff files between the C and O backups.  Then since the previous full copy can be recreated from the diff files that have been made, the C and O backups on the server will be rsynced to be identical.  After that is completed the remote utility will rsync to the (C) backup on the server.
4) We will address backup compression if a future iteration, not yet.
5) A .config file located in the same place as the linux daemon executable file contains the base location for the backups, it will default to /blzbak at installation time or if no .config is found.
6) YAML
7) Snapshots will be configurable, daily is default, but could be whatever is possible in  CRON.  There will be a .config file for the remote command line utility also stored in same place as executable.
8) Yes lets use same patterns as gitignore and put the file .blzbak-ignore in the same location as the executable.  Note that config and ignore setting will be stored in each backup set so that future changes to the .config and .blzbak-ignore files will not affect existing backup sets.  
9) Restorations will default to the original location unless specified otherwise.  Prompt if not specified, a simple enter key for original or the alternate base location will be entered at the prompt unless specified by command line option.
10) The GUI will allow for the browsing of available snapshots.  All the backup functions, listing of backup sets, backup instances, and folders files can be done through the command line utility as well.  The GUI will perform all its functions by calling this utility with command line arguments and an optional window in the GUI that can be opened to peek in on what is happening behind the scenes with the calls to the command line utility.
11) The CLI must communicate with the with the daemon, one to get lists of existing backups when needed, and also as described before when it needs to prepare for a new backup about to be performed.  The daemon will have one thread for listening for the client (remote) messages and another thread to perform the DIFF required before a new fsync is run from the client.  The first thing CLI does when being run is establish a communication channel with the daemon on the server.

**Project structure** (`blzbak/blzbak/`):

| File | Purpose |
|---|---|
| pyproject.toml | Build config; `pyyaml` + `pathspec` deps; blzbak console script |
| protocol.py | Length-prefixed JSON framing for daemon TCP comms |
| models.py | `BackupSet`, `ServerConfig`, `Snapshot`; name validation prevents path traversal |
| config.py | Loads `.config` / `.blzbak-ignore` from the executable's directory; safe YAML deep-merge |
| client.py | `DaemonClient` — context-manager TCP client; extended 300 s timeout for `PREPARE_BACKUP` |
| ignore.py | gitignore-style patterns via `pathspec`; converts to rsync `--exclude=` args |
| rsync.py | `build_backup_cmd` / `build_restore_cmd` / `run_rsync`; `--relative` + `/./<path>` trick for restores |
| cron.py | `# blzbak-managed:<name>` tagged cron entries; idempotent install |
| backup.py | YAML backup-set persistence in `sets_directory` |
| commands | `set_cmd`, `backup_cmd`, `restore_cmd`, `cron_cmd` |
| cli.py | argparse entry point; daemon connected only for `backup`/`restore` commands |

**Key design decisions implemented:**
- Backup set configs are **self-contained** — ignore patterns and server info are snapshotted at creation time
- `PREPARE_BACKUP` → daemon diffs C/O and syncs → CLI rsyncs to C → `BACKUP_COMPLETE`
- Restores use rsync `--relative` with the `/./<path>` trick so original paths are always preserved
- All subprocess calls use list args (`shell=False`) — no injection risk

**Storage structure:**
/blzbak/                      (configurable base_path)
├── <set_name>/
│   ├── C/                    (current backup)
│   ├── O/                    (once-removed backup)
│   └── metadata.yaml
└── diffs/
    └── <set_name>/
        └── diff_<timestamp>.tar.gz

**Backup workflow:**
1. Client sends PREPARE_BACKUP command
2. Daemon creates diff between C and O snapshots
3. Daemon syncs O to match C (rsync --delete)
4. Client rsyncs new backup to C directory
5. Client sends BACKUP_COMPLETE notification
